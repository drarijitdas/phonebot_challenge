## Spacing Scale

Rich Console output does not use pixel spacing. The spacing contract is expressed in terms of **blank lines** and **indentation levels** in terminal output.

| Token | Value | Usage |
|-------|-------|-------|
| inline | 0 blank lines | Same-group message continuation |
| sm | 1 blank line | Between logical output sections |
| md | 2 blank lines | Before and after Rich tables |
| separator | `\n` + `---` rule | Between startup banner and pipeline body |

Exceptions: Phoenix URL line is printed with 1 blank line above it to visually separate it from the accuracy table. Trace count line is printed immediately below Phoenix URL with no blank line between.

> Source: run.py established pattern (console.print with blank lines between sections)

## Typography

Rich Console typography is expressed as **markup weight and style**, not pixel sizes.

| Role | Markup | Weight Equivalent | Usage |
|------|--------|------------------|-------|
| Body | plain text | regular (400) | Progress messages, file paths, counts |
| Label | `[bold]` | semibold (600) | Section headings, table column headers |
| Heading | `[bold green]` | semibold (600) | Phase-level status announcements |
| Monospace value | `[cyan]` | regular (400) | Recording IDs, field names in tables |

Line height: single-spaced (terminal default). No line-height overrides.

> Source: run.py established pattern — `[bold green]` for startup, `[bold]` for table headers, `[cyan]` for field names, `[green]` for success values

## Color

Rich Console uses named color tokens, not hex values. The 60/30/10 rule maps to output density.

| Role | Rich Token | Usage |
|------|-----------|-------|
| Dominant (60%) | default (white/gray terminal fg) | Body text, progress messages, file paths |
| Secondary (30%) | `[cyan]` | Recording IDs, field names, config values printed at startup |
| Accent (10%) | `[green]` | Success states, completion messages, accuracy values |
| Warning | `[yellow]` | Non-fatal notices (e.g. Phoenix already running on port) |
| Destructive | `[red]` | Errors and fatal failure messages only |

Accent (`[green]`) reserved for:
- Pipeline completion confirmation line ("Extraction complete in X.Xs")
- Phoenix URL confirmation line ("Phoenix UI: http://localhost:6006")
- Trace count confirmation line ("30 traces sent to Phoenix")
- Overall accuracy row in Per-Field Accuracy table

`[red]` reserved for:
- Missing transcript error ("No transcripts found...")
- Phoenix startup failure
- Any unhandled exception surfaced to console

> Source: run.py established patterns; D-03, D-14 (CONTEXT.md) for Phoenix URL and trace count messages

## Interaction Flow Contract

Phase 4 adds three new interaction beats to the existing run.py startup/completion flow. The full ordered sequence must be:

```
1. [startup]   Phonebot pipeline starting...
               Model: {model}
               Prompt version: {version}          ← NEW in Phase 4 (D-06)
               Recordings: {dir}
               Output: {path}

2. [tracing]   Tracing initialized — project: {project_name}   ← NEW (D-09, D-11)

3. [pipeline]  Extracting from 30 recordings...
               Extraction complete in X.Xs
               Results written to outputs/results.json

4. [accuracy]  [Rich table: Per-Field Accuracy]

5. [phoenix]   30 traces sent to Phoenix           ← NEW (D-14)
               Phoenix UI: http://localhost:6006    ← NEW (D-03)
```

Step 2 (tracing init) must appear BEFORE step 3 (pipeline) — init_tracing() is called before run_pipeline().
Steps 5a and 5b appear AFTER the accuracy table — Phoenix confirmation is the last output.

> Source: D-01 (auto-start), D-03 (URL print), D-06 (--prompt-version flag), D-09 (init_tracing module), D-14 (trace count); run.py for existing step ordering

## CLI Argument Contract

Phase 4 adds one new argument to run.py's argparse parser.

| Argument | Type | Default | Help text |
|----------|------|---------|-----------|
| `--prompt-version` | str | `"v1"` | Prompt version tag attached to every Phoenix trace |

Existing arguments (unchanged): `--model`, `--recordings-dir`, `--output`

`--prompt-version` is printed in the config echo block (step 1) and passed to init_tracing() as trace metadata.

> Source: D-06 (CONTEXT.md); run.py argparse established pattern
